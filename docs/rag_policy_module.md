# 本地 RAG 售后政策模块

本项目新增了一个轻量级本地 RAG 模块，用于让客服回复和售后策略参考明确的业务政策，而不是完全依赖大模型自由生成。

## 为什么加 RAG

原来的流程可以判断情绪、分类和风险，但客服回复主要依赖模型生成。这样会有两个问题：

1. 回复可能缺少明确政策依据。
2. 模型可能编造平台规则或赔偿承诺。

加入 RAG 后，系统会先从本地售后政策库和客服话术库中检索相关内容，再把检索结果交给回复生成节点。

## 当前实现

知识库文件：

```text
data/knowledge/after_sales_policy.md
data/knowledge/reply_templates.md
```

RAG 代码：

```text
src/ecommerce_agent/rag/knowledge_base.py
src/ecommerce_agent/rag/policy_rag.py
```

接入位置：

```text
输入理解
-> 情感分析
-> 问题分类
-> 风险判断
-> RAG 售后政策检索
-> 售后策略
-> 客服回复
-> 汇总结果
```

## 检索方式

当前版本使用本地关键词检索，不依赖外部向量数据库。

检索信号来自：

- 用户原文
- 问题分类
- 二级原因
- 命中关键词
- 风险等级

例如用户说：

```text
耳机用了三天听筒就坏了，我要退货
```

系统会检索到：

```text
P002 质量问题退换货
P006 高风险投诉升级
T001 质量问题安抚话术
T004 退款退货话术
```

## 为什么第一版不用向量数据库

本项目第一版 RAG 的知识库规模较小，售后政策和话术条目较明确。使用本地关键词检索有几个好处：

1. 不需要额外安装 Chroma/FAISS。
2. 不需要下载 embedding 模型。
3. 更容易解释和调试。
4. 更适合先证明业务流程。

后续如果知识库变大，可以升级为：

```text
Chroma 本地向量库
BM25 + 向量混合检索
重排序 reranker
政策引用追踪
```

## 模块说明

可以这样介绍：

我在 Agent 流程中加入了本地 RAG 售后政策检索节点。系统在完成情感分析、问题分类和风险判断后，会根据分类结果、风险等级和关键词检索售后政策与客服话术，再把检索到的依据传给回复生成节点。这样客服回复不是单纯由大模型自由生成，而是受平台规则约束，降低了幻觉和错误承诺风险。
