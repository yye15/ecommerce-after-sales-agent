# 系统架构说明

## 设计目标

系统目标是把“客户反馈文本”转成“客服和运营能直接使用的动作”。所以流程不是单次问答，而是多节点协作：

1. 输入理解：清洗文本，判断是反馈、投诉还是售后请求。
2. 情感分析：抽取方面词、评价词和情感极性。
3. 问题分类：把三元组映射到业务问题。
4. 风险判断：识别投诉、退款、维权等高风险信号。
5. 售后策略：生成处理动作和负责人。
6. 回复生成：输出可直接发给客户的话术。
7. 运营分析：对批量评论做问题统计。

## 为什么用 LangGraph

LangGraph 适合表达这种业务流程，因为每一步都有明确状态输入和输出：

```text
State = {
  raw_text,
  sentiment,
  issue_categories,
  risk,
  strategy,
  reply
}
```

普通网页 AI 更像一次问答，而这个项目可以把中间结果保留下来，并根据风险等级走不同路径。

## LALUN 的位置

LALUN 对应的是系统里的“细粒度情感分析节点”。当前版本：

- 中文场景：DeepSeek + 规则兜底
- LALUN：先做本地权重检测和适配层

未来版本：

```text
中文评论 -> 中文预处理 -> LALUN 推理 -> 情感三元组 -> 后续 Agent 流程
```

这能把你的论文成果转成一个产品型能力。
