# Demo Cases

本文件用于项目展示和功能复现。每个案例都对应一个典型电商售后场景，展示系统如何从用户文本中提取问题、判断风险、检索政策并生成客服回复。

复现方式：

```powershell
python main.py --no-llm --text "你的测试文本"
```

说明：以下案例使用离线规则兜底模式 `--no-llm` 生成，便于稳定复现。实际开启 DeepSeek API 后，情感抽取和客服回复会更自然，但整体流程和业务判断逻辑保持一致。

## Case 1：质量故障 + 退货诉求 + 客服不处理

**输入文本**

```text
耳机用了三天听筒就坏了，客服一直不处理，我要退货。
```

**业务关注点**

验证系统能识别商品质量问题、客服响应问题和退货诉求，并检索质量退换货政策。

**情感三元组**

| aspect | opinion | sentiment |
|---|---|---|
| 客服 | 退货 | NEG |
| 商品质量 | 坏 | NEG |
| 退换货 | 退货 | NEG |

**问题分类**

| 一级类别 | 业务维度 | 二级原因 |
|---|---|---|
| 客服响应问题 | 服务 | 响应慢或失联、处理拖延 |
| 商品质量问题 | 商品 | 核心故障、耐用性不足 |
| 退换货与赔付问题 | 售后 | 退货/换货诉求 |

**风险判断**

```text
风险等级：中风险
优先级：P2
风险分数：81
```

主要依据：

- 识别到 3 个负面情感点。
- 听筒坏属于核心功能/商品质量问题。
- 同一条反馈同时涉及质量、客服响应和退货诉求，存在升级风险。

**RAG 检索到的政策**

```text
P002 质量问题退换货
T001 质量问题安抚话术
T004 退款退货话术
P004 客服响应与人工升级
```

**售后策略**

```text
1. 升级客服工单：检查历史响应记录，承诺明确处理时限。
2. 收集凭证：引导客户提供照片、视频或订单号。
3. 核验售后政策：核对订单状态、签收时间和商品问题。
```

**客服回复**

```text
您好，非常抱歉给您带来不便。关于您提到的客服响应问题、商品质量问题、退换货与赔付问题，检查历史响应记录，承诺明确处理时限，避免继续无人跟进。我们会参考《P002 质量问题退换货》为您核实处理。请您提供订单号，我们会尽快给出明确处理结果。
```

**展示讲解点**

这个案例适合说明：系统不是简单识别负面情绪，而是能把“听筒坏”“客服不处理”“我要退货”拆成质量、服务和售后三类问题，并用 RAG 检索质量退换货政策，生成有依据的处理回复。

## Case 2：物流延迟 + 客服不回复 + 投诉倾向

**输入文本**

```text
这个耳机音质不错，但是物流太慢，客服也一直不回复，我有点想投诉。
```

**业务关注点**

验证系统能处理混合情绪评论：既有正面商品体验，也有负面物流和客服反馈，并能识别投诉倾向。

**情感三元组**

| aspect | opinion | sentiment |
|---|---|---|
| 物流 | 太慢 | NEG |
| 客服 | 不回复 | NEG |
| 音质 | 不错 | POS |

**问题分类**

| 一级类别 | 业务维度 | 二级原因 |
|---|---|---|
| 物流问题 | 履约 | 配送异常 |
| 客服响应问题 | 服务 | 响应慢或失联 |
| 产品体验问题 | 体验 | 核心功能体验 |

**风险判断**

```text
风险等级：高风险
优先级：P1
风险分数：100
是否人工升级：是
```

主要依据：

- 出现“投诉”信号。
- 物流慢和客服不回复同时出现。
- 同一条反馈涉及多个问题，存在升级风险。

**RAG 检索到的政策**

```text
P006 高风险投诉升级
P003 物流延迟与配送异常
T002 高风险投诉升级话术
P004 客服响应与人工升级
```

**售后策略**

```text
1. 人工优先介入：客服主管优先处理。
2. 查询物流：核实物流节点和预计送达时间。
3. 升级客服工单：检查历史响应记录。
4. 收集凭证：补充订单号和沟通记录。
```

**客服回复**

```text
您好，非常抱歉给您带来不好的体验，我们已将您的问题优先升级处理。关于您提到的物流问题、客服响应问题、产品体验问题，建议客服主管优先介入。我们会参考《P006 高风险投诉升级》为您核实处理。请您提供订单号，我们会尽快给出明确处理结果。
```

**展示讲解点**

这个案例最适合展示项目亮点：系统不会因为“音质不错”而忽略后面的投诉风险，而是能识别混合情绪，并把“物流太慢 + 客服不回复 + 投诉倾向”升级为 P1 高风险。

## Case 3：温和体验反馈

**输入文本**

```text
屏幕的显示效果有待提高，10寸还是小了，不是那种很抢人眼球的感觉。
```

**业务关注点**

验证系统不会把轻微体验反馈过度升级，同时仍能保留产品体验问题分类。

**情感三元组**

| aspect | opinion | sentiment |
|---|---|---|
| 屏幕 | 提及 | NEU |

**问题分类**

| 一级类别 | 业务维度 | 二级原因 |
|---|---|---|
| 产品体验问题 | 体验 | 内容/图片不符、核心功能体验 |

**风险判断**

```text
风险等级：低风险
优先级：P3
风险分数：44
```

主要依据：

- 文本只是轻微体验不满，没有退款、投诉、故障、安全隐患等高风险信号。
- 系统保留“产品体验问题”分类，但通过轻微体验降级规则避免过度升级。

**RAG 检索到的政策**

```text
T006 温和体验反馈话术
P002 质量问题退换货
P003 物流延迟与配送异常
T001 质量问题安抚话术
```

**售后策略**

```text
1. 收集凭证：如果用户后续需要售后，可引导补充照片、视频或订单号。
```

**客服回复**

```text
您好，非常抱歉给您带来不便。关于您提到的产品体验问题，引导客户提供照片、视频或订单号，用于判断退换货或补发。我们会参考《T006 温和体验反馈话术》为您核实处理。请您提供订单号，我们会尽快给出明确处理结果。
```

**展示讲解点**

这个案例可以说明你做了“风险校准”，不是所有负面体验都直接升高风险。系统能识别体验问题，但不会把温和反馈误判成高优先级投诉。

## Case 4：价格权益争议

**输入文本**

```text
价格比我买的时候便宜了好多，感觉被坑了，赠品也没有发。
```

**业务关注点**

验证系统能识别价格差异、赠品缺失和权益争议，并检索价格权益相关政策。

**情感三元组**

| aspect | opinion | sentiment |
|---|---|---|
| 价格 | 被坑 | NEG |

**问题分类**

| 一级类别 | 业务维度 | 二级原因 |
|---|---|---|
| 价格与权益问题 | 权益 | 价格争议、优惠/赠品争议 |

**风险判断**

```text
风险等级：中风险
优先级：P2
风险分数：76
```

主要依据：

- “被坑”属于交易信任不满信号。
- 同时涉及价格变化和赠品未发，容易引发投诉或差评。

**RAG 检索到的政策**

```text
T005 价格权益话术
P005 价格、优惠与权益争议
T004 退款退货话术
T001 质量问题安抚话术
```

**售后策略**

```text
1. 核对权益：核实优惠券、差价保护和活动规则，避免模糊解释引发投诉。
```

**客服回复**

```text
您好，非常抱歉给您带来不便。关于您提到的价格与权益问题，核实优惠券、差价保护、活动规则，避免模糊解释引发投诉。我们会参考《T005 价格权益话术》为您核实处理。请您提供订单号，我们会尽快给出明确处理结果。
```

**展示讲解点**

这个案例可以说明系统不仅处理质量和物流，也能处理价格、优惠、赠品等权益问题，覆盖了真实电商售后中的常见争议。

## Case 5：正向反馈

**输入文本**

```text
衣服质量很好，尺码也合适，物流也很快，下次还会买。
```

**业务关注点**

验证系统对正向反馈保持低风险，并生成感谢型回复。

**情感三元组**

| aspect | opinion | sentiment |
|---|---|---|
| 物流 | 快 | POS |
| 商品质量 | 很好 | POS |
| 尺码 | 好 | POS |

**问题分类**

```text
其他反馈
```

**风险判断**

```text
风险等级：低风险
优先级：P3
风险分数：20
```

主要依据：

- 未发现明显风险信号。
- 情感点均为正向。

**RAG 检索到的政策**

```text
T006 温和体验反馈话术
P003 物流延迟与配送异常
P002 质量问题退换货
P004 客服响应与人工升级
```

**客服回复**

```text
您好，感谢您的认可和反馈。关于您提到的其他反馈，感谢客户反馈，继续追问订单号和具体问题。我们会参考《T006 温和体验反馈话术》为您核实处理。请您提供订单号，我们会尽快给出明确处理结果。
```

**展示讲解点**

这个案例说明系统不会只关注负面反馈，也能识别正向评价并保持低风险。但当前正向反馈的回复仍偏售后化，后续可以单独增加“正向评价感谢/复购引导”策略，让回复更自然。

## 推荐演示顺序

如果展示时间有限，建议优先展示：

1. **Case 2：物流延迟 + 客服不回复 + 投诉倾向**  
   最能展示高风险识别、人工升级和 RAG 政策检索。

2. **Case 1：质量故障 + 退货诉求 + 客服不处理**  
   展示细粒度问题归因和售后策略。

3. **Case 3：温和体验反馈**  
   展示风险校准能力，说明系统不会一刀切升级所有负面评论。

4. **Case 4：价格权益争议**  
   展示业务覆盖面。

5. **Case 5：正向反馈**  
   展示系统边界和后续优化空间。

## Demo 总讲法

可以这样介绍这些 Demo：

```text
我准备了几类典型售后场景，包括质量故障、物流延迟、客服不回复、价格权益争议和温和体验反馈。每个案例都会经过同一条 LangGraph 流程：先做情感三元组抽取，再做问题分类和风险判断，然后检索本地售后政策库，最后生成售后策略和客服回复。

这些案例的重点不是让回复看起来像聊天，而是验证系统能不能把客户文本转成结构化业务判断，例如什么问题、什么风险、要不要升级、依据哪条政策处理。
```
